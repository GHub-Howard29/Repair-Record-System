## 1. Google OAuth Web Client

OAuth Client 類型應選：

```text
Web application
```

`.env` 欄位：

```env
VITE_GOOGLE_CLIENT_ID=你的ClientID.apps.googleusercontent.com
```

已授權的 JavaScript 來源：

```text
http://127.0.0.1:5173
http://localhost:5173
https://repair-record-system.firebaseapp.com
https://ghub-howard29.github.io
```

注意：

- GitHub Pages 專案頁只填 origin，不填 `/Repair-Record-System/` path。
- 目前不需要填已授權的重新導向 URI。
- `.env` 不可提交到 Git。

驗證方式：

1. 填入 `VITE_GOOGLE_CLIENT_ID`。
2. 重啟 Vite dev server。
3. 開啟 `http://127.0.0.1:5173/Repair-Record-System/`。
4. 登入頁應顯示 Google 官方登入按鈕，而不是本機開發登入。

若看到 `Google 登入已略過：unknown_reason`，通常是 One Tap prompt 被瀏覽器或 Google 狀態略過。系統已改用 Google 官方登入按鈕 `renderButton()`，不再依賴 One Tap 自動提示。

## 3. Firebase / Firestore

用途：

- 保存維修紀錄文字資料。
- 支援跨裝置即時查看。
- 後續替換目前 `localStorage` repository。

需要建立：

1. Firebase project。
2. Web app。
3. Firestore database。
4. Firebase Authentication 的 Google sign-in provider。
5. Firestore security rules。

`.env` 欄位：

```env
VITE_FIREBASE_API_KEY=
VITE_FIREBASE_AUTH_DOMAIN=
VITE_FIREBASE_PROJECT_ID=
VITE_FIREBASE_APP_ID=
```

Firebase Console 取得位置：

```text
Project settings → General → Your apps → Web app → Firebase SDK snippet → Config
```

欄位對應：

| Firebase config | `.env` |
|-----------------|--------|
| `apiKey` | `VITE_FIREBASE_API_KEY` |
| `authDomain` | `VITE_FIREBASE_AUTH_DOMAIN` |
| `projectId` | `VITE_FIREBASE_PROJECT_ID` |
| `appId` | `VITE_FIREBASE_APP_ID` |
| `storageBucket` | 目前不使用，附件改由 Google Drive 保存 |
| `messagingSenderId` | 目前不使用 |
| `measurementId` | 目前不使用 |

建議 Firestore collection：

```text
repairRecords
```

文件 ID：

```text
RepairRecord.id
```

第一版資料權限建議：

- 必須透過 Firebase Auth 登入後才能讀寫。
- Google OAuth 登入成功後，前端會同步建立 Firebase Auth session，讓 Firestore Rules 可使用 `request.auth` 判斷身分。
- 第一版單一使用者，不做角色權限。
- 不提供刪除維修紀錄。

Rules 範本：

```text
firestore.rules
```

已由使用者在 Firebase Console 更新 Firestore Rules。後續若新增使用者，需將該 Google 帳號 Email 加入 allowlist；未列入清單的帳號不可讀寫維修紀錄。

待程式串接位置：

```text
src/services/firestoreRepairRecordService.ts
```

## 4. Google Drive API

用途：

- 保存照片附件。
- 後續透過後端代理替換目前附件 data URL 暫存。

需要建立：

1. 啟用 Google Drive API。
2. 建立專用資料夾，例如 `Repair Record Attachments`。
3. 取得資料夾 ID。
4. 後續建立 Service Account，並將資料夾分享給該 Service Account email。

`.env` 欄位：

```env
VITE_GOOGLE_DRIVE_FOLDER_ID=
VITE_GOOGLE_DRIVE_SCOPE=https://www.googleapis.com/auth/drive.file
```

目前保留的前端 scope：

```text
https://www.googleapis.com/auth/drive.file
```

待程式串接位置：

```text
src/services/googleDriveAttachmentService.ts
```

注意：

- 目前 Google 登入只取得登入身分。
- 不將 Google Service Account private key 放入前端 `.env` 或任何 `VITE_*` 欄位。
- 正式 Drive 上傳需由 Firebase Functions 或等效後端 endpoint 代理處理。
- 附件仍需遵守完成後鎖定，不可新增、更換、刪除。

## Google Drive Service Account 資安決策

本專案不會把 Google Service Account credentials 放入 Vite 前端 `.env`。

原因：

- 目前 app 是 GitHub Pages 靜態前端。
- 任何 `VITE_*` 變數在 build 後都可被瀏覽器端 JavaScript 讀到。
- Service Account private key 必須視為後端 secret。

建議 Drive 上傳架構：

```text
Frontend -> Firebase Function upload endpoint -> Google Drive API -> Drive folder
```

前端後續只可新增非敏感設定，例如：

```env
VITE_ATTACHMENT_UPLOAD_MODE=service-account
VITE_ATTACHMENT_UPLOAD_ENDPOINT=https://your-function-url.example/upload
```

後端專用設定才可存放敏感值：

```env
GOOGLE_SERVICE_ACCOUNT_EMAIL=
GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY=
GOOGLE_DRIVE_FOLDER_ID=
```

實作注意事項：

1. 將 Google Drive 目標資料夾分享給 Service Account email。
2. Service Account credentials 存在 Firebase Functions config、Secret Manager 或等效後端 secrets。
3. `VITE_GOOGLE_DRIVE_FOLDER_ID` 只作為暫時或非敏感前端設定；後端 upload flow 完成後，Drive 寫入權限以後端設定為準。
4. 後端 upload flow 建立後，更新 `src/services/googleDriveAttachmentService.ts`，改呼叫後端 endpoint，不從瀏覽器直接呼叫 Google Drive 寫入 API。
