## 1. Google OAuth Web Client

OAuth Client 類型應選：

```text
Web application
```

`.env` 欄位：

```env
VITE_GOOGLE_CLIENT_ID=你的ClientID.apps.googleusercontent.com
```

已授權的 JavaScript 來源：

```text
http://127.0.0.1:5173
http://localhost:5173
https://ghub-howard29.github.io
```

注意：

- GitHub Pages 專案頁只填 origin，不填 `/Repair-Record-System/` path。
- 目前不需要填已授權的重新導向 URI。
- `.env` 不可提交到 Git。

驗證方式：

1. 填入 `VITE_GOOGLE_CLIENT_ID`。
2. 重啟 Vite dev server。
3. 開啟 `http://127.0.0.1:5173/Repair-Record-System/`。
4. 登入頁應顯示 Google 官方登入按鈕，而不是本機開發登入。

若看到 `Google 登入已略過：unknown_reason`，通常是 One Tap prompt 被瀏覽器或 Google 狀態略過。系統已改用 Google 官方登入按鈕 `renderButton()`，不再依賴 One Tap 自動提示。

## 2. GitHub Pages

目前專案固定部署到 GitHub Pages 專案頁：

```text
https://ghub-howard29.github.io/Repair-Record-System/
```

已設定：

```ts
base: '/Repair-Record-System/'
```

位置：

```text
vite.config.ts
```

注意：

- `base` 是專案設定一次，不需要每次部署手動設定。
- PWA manifest、favicon、service worker 已調整為跟隨 Vite base。

已建立 GitHub Actions workflow：

```text
.github/workflows/deploy-pages.yml
```

GitHub repository settings 需確認：

1. `Settings` → `Pages`
2. `Build and deployment`
3. `Source` 選 `GitHub Actions`

GitHub Actions secrets 建議建立：

```text
VITE_GOOGLE_CLIENT_ID
VITE_FIREBASE_API_KEY
VITE_FIREBASE_AUTH_DOMAIN
VITE_FIREBASE_PROJECT_ID
VITE_FIREBASE_APP_ID
VITE_GOOGLE_DRIVE_FOLDER_ID
```

目前已需要的 secret：

```text
VITE_GOOGLE_CLIENT_ID
```

其餘 Firebase / Drive 欄位可先建立空值或等串接時再補。

部署驗證方式：

1. push 到 `main`。
2. 到 GitHub `Actions` 查看 `Deploy GitHub Pages` 是否成功。
3. 開啟 `https://ghub-howard29.github.io/Repair-Record-System/`。
4. 確認頁面、JS/CSS、manifest、Google 登入皆正常。

PWA / 網頁圖示：

```text
public/repair-system-icon-sheet.png
public/pwa-icon-512.png
public/pwa-icon-192.png
public/apple-touch-icon.png
public/favicon-32.png
```

其中 `repair-system-icon-sheet.png` 是原始功能圖示素材，其他檔案為裁切後輸出，已接到 manifest、favicon、Apple touch icon 與分享縮圖。

## 3. Firebase / Firestore

用途：

- 保存維修紀錄文字資料。
- 支援跨裝置即時查看。
- 後續替換目前 `localStorage` repository。

需要建立：

1. Firebase project。
2. Web app。
3. Firestore database。
4. Firestore security rules。

`.env` 欄位：

```env
VITE_FIREBASE_API_KEY=
VITE_FIREBASE_AUTH_DOMAIN=
VITE_FIREBASE_PROJECT_ID=
VITE_FIREBASE_APP_ID=
```

Firebase Console 取得位置：

```text
Project settings → General → Your apps → Web app → Firebase SDK snippet → Config
```

欄位對應：

| Firebase config | `.env` |
|-----------------|--------|
| `apiKey` | `VITE_FIREBASE_API_KEY` |
| `authDomain` | `VITE_FIREBASE_AUTH_DOMAIN` |
| `projectId` | `VITE_FIREBASE_PROJECT_ID` |
| `appId` | `VITE_FIREBASE_APP_ID` |
| `storageBucket` | 目前不使用，附件改由 Google Drive 保存 |
| `messagingSenderId` | 目前不使用 |
| `measurementId` | 目前不使用 |

建議 Firestore collection：

```text
repairRecords
```

文件 ID：

```text
RepairRecord.id
```

第一版資料權限建議：

- 必須登入後才能讀寫。
- 第一版單一使用者，不做角色權限。
- 不提供刪除維修紀錄。

待程式串接位置：

```text
src/services/firestoreRepairRecordService.ts
```

## 4. Google Drive API

用途：

- 保存照片附件。
- 後續替換目前附件 data URL 暫存。

需要建立：

1. 啟用 Google Drive API。
2. 建立專用資料夾，例如 `Repair Record Attachments`。
3. 取得資料夾 ID。
4. 在 OAuth consent screen 加入 Drive 相關 scope。

`.env` 欄位：

```env
VITE_GOOGLE_DRIVE_FOLDER_ID=
VITE_GOOGLE_DRIVE_SCOPE=https://www.googleapis.com/auth/drive.file
```

建議 scope：

```text
https://www.googleapis.com/auth/drive.file
```

待程式串接位置：

```text
src/services/googleDriveAttachmentService.ts
```

注意：

- 目前 Google 登入只取得登入身分。
- Drive 上傳需要額外取得 access token 與 Drive scope。
- 附件仍需遵守完成後鎖定，不可新增、更換、刪除。

## 5. 部署流程

已建立：

- GitHub Actions 自動部署。
- 本機部署前檢查指令：

```powershell
npm run deploy:check
```

建議順序：

1. 先確認 `npm run build` 可成功。
2. push 到 `main` 觸發 GitHub Actions。
3. 部署後確認：
   - 頁面能開啟。
   - JS / CSS 資源路徑正常。
   - Google 登入來源已被允許。
   - PWA manifest 可讀取。
