# 04_工作規範

## 目的

本文件定義維修紀錄系統後續開發規範。所有工作應依 `docs/01_需求分析.md`、`docs/02_系統架構.md`、`docs/03_開發 Roadmap.md` 與 `src/agent.ts` 執行。

## 開發流程

1. 每次開工前先確認目前 Phase 與交接紀錄。
2. 優先維護核心業務規則，不以 UI 便利性破壞資料完整性。
3. 每個 Phase 完成後需完成建置或測試、更新文件、建立 Git Commit。
4. 未完成項目必須寫入 `docs/05_交接紀錄.md`。

## 環境設定

- `VITE_GOOGLE_CLIENT_ID`：Google OAuth Client ID。未設定時系統會使用本機開發登入，僅供開發流程驗證。
- 正式部署前必須設定 Google OAuth，且不得依賴本機開發登入作為正式驗收。
- GitHub Pages OAuth JavaScript origin 只填網域，例如 `https://你的GitHub帳號.github.io`，不要填 `/Repair-Record-System/` 路徑。
- 本專案部署在 GitHub Pages 專案頁，`vite.config.ts` 的 `base` 固定為 `/Repair-Record-System/`，不需每次部署手動設定。

## 程式架構規範

- `src/auth`：登入、登出、登入狀態。
- `src/features/repair`：維修紀錄核心規則、建立、編輯、完成鎖定。
- `src/features/attachment`：照片壓縮、預覽、上傳與附件異動規則。
- `src/features/search`：搜尋、篩選、歷史維修摘要。
- `src/features/warranty`：保固判斷。
- `src/features/export`：PDF 與 Excel 匯出。
- `src/features/sync`：文字與附件同步、離線暫存、待同步佇列。
- `src/storage`：本機資料存取與離線資料保留。
- `src/types`：跨模組共用型別。

## 核心規則

- 不建立刪除維修紀錄功能。
- 送回日期一旦填寫，案件與附件皆不可再修改。
- 同一製造號碼不得同時存在兩筆未完成案件。
- 建立案件至少需要收到日期、回送地點、製造號碼。
- 文字資料優先同步；附件依 Wi-Fi、行動網路、離線狀態分流。
- 同步失敗不得遺失資料，必須保留待同步狀態。

## 驗收規範

- 新增或修改核心規則時，需補對應測試或在交接紀錄留下人工驗證結果。
- UI 必須能在桌面與手機寬度操作。
- 完成案件的表單與附件操作必須呈現唯讀或停用狀態。
- 匯出資料必須與系統資料一致。
